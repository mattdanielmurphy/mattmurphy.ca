## Goal
Implement a push-based GitHub Webhook architecture for revalidating notes, replacing the previous background polling mechanism that risked hitting the GitHub API rate limit (5,000 requests/hour) due to fetching all markdown files on every interval.

## Changes Made
- Added `src/pages/api/revalidate-notes.js`: Endpoint for handling incoming GitHub Webhook push events. It verifies `req.query.secret`, triggers `res.revalidate('/notes')`, and parses `req.body.commits` to specifically revalidate individual modified notes (e.g. `/notes/slug`).
- Modified `src/pages/notes/index.js`: Removed `revalidate: 300` to stop background polling.
- Modified `src/pages/notes/[...slug].js`: Removed `revalidate: 60` to stop background polling.

## What Worked
- Replaced the polling intervals successfully.
- Webhook logic handles specific paths correctly (removing the `.md` extension and encoding the URI).

## What Didn't Work / Known Issues
- Currently relies on the user setting up the `REVALIDATE_SECRET` in Vercel and linking the webhook manually on GitHub. If they fail to do so, notes won't update.

## Architecture Notes
- The site now uses On-Demand Revalidation (Next.js Pages Router). It assumes `REVALIDATE_SECRET` environment variable exists. The API URL must be queried using `?secret=YOUR_SECRET`.
